# Verification

The peephole replaces the `balanceOf` slot function ‚Äî a keccak hash ‚Äî with the
bitwise complement `~addr`. That is sound **iff the replacement is injective**;
the theorems below establish it.

## In this repo (self-contained, mathlib-free)

[`VenomOpt/Peephole.lean`](VenomOpt/Peephole.lean) proves the mathematical core
on `BitVec 256` (standard axioms only):

| theorem | statement |
|---|---|
| `optSlot_injective` | `~¬∑` is injective (it is its own inverse) |
| `distinct_addresses_distinct_opt_slots` | `a ‚âÝ b ‚Üí ~a ‚âÝ ~b` ‚Äî no slot collisions |
| `write_no_alias` | writing holder `a`'s balance never changes holder `b`'s (`b ‚âÝ a`) |
| `read_own_write` | a holder reads back exactly what was written to their slot |
| `unpatched_read_misses_patched_write` | **completeness matters**: a write via `~key` is invisible to a read still deriving the slot any other way ‚Äî a *partial* patch splits the map, so the patcher must rewrite every site (the differential tests are the completeness oracle) |

Build & check:

```bash
cd verification && lake build      # builds in seconds ‚Äî no mathlib
```

## The full, end-to-end statement (EVMYulLean)

The proof here is the *slot-function* argument. That the **rewritten EVM
bytecode** computes the same storage as the original ‚Äî against EVMYulLean's real
`EVM.step` ‚Äî is proved in the EVMYulLean development:

| EVMYulLean result | what it adds |
|---|---|
| `NoAlias.distinct_addresses_distinct_opt_slots` | the same injectivity, on `UInt256.lnot` |
| `SlotAbstraction.realizes_write_opt` | the `~addr` storage scheme realizes balance write-through **unconditionally** |
| `SlotAbstraction.write_opt_preserves_named` | a `~addr` write never disturbs a named low slot (`totalSupply`, ‚Ä¶) |
| `BalanceSlot.venomBalanceLoad_orig_opt_equiv` | original (keccak) and optimized (`~addr`) loads agree under the storage relation |
| `Solvency.transfer_preserves_solvent` / `Erc20.*` | the rewrite preserves `Œ£ balances = totalSupply` and the ERC-20 spec |

The rewrite operates on Venom IR, which is then compiled to EVM. That the
compiler step itself is sound ‚Äî the **generated EVM program simulates the
source Venom function** ‚Äî is proved in EVMYulLean's Venom‚ÜíEVM codegen
development, so the compiled output of the (optimized) Venom is correct too:

| EVMYulLean result (`EvmYul/Venom/Hol/Codegen/`) | what it adds |
|---|---|
| `CodegenCorrectness.codegen_fn_correct` / `codegen_correct` | the generated program simulates the source Venom function / context |
| `CodegenCorrectness.codegenFuel_correct_nonvacuous_witness` | ‚Ä¶non-vacuously ‚Äî a real generated program, not an empty/vacuous witness |
| `GenBlockSim.genBlockSimulation` | the per-block simulation capstone the above rests on |
| `GenInstSim.genRegularInstPlan_log_sim` | end-to-end sim for a variable-arity instruction (LOG), incl. the fixed operand order |

The differential harness drives **dynamic-ABI** calldata (DynArray / Bytes /
struct) generated by evm-abi-lean's verified encoder. That those byte layouts
are the right ones ‚Äî and that the rewrite is invisible at the ABI-call level ‚Äî
is pinned by two more maps:

| result | what it adds |
|---|---|
| `AbiCrossval.erc20_selectors_match_keccak` (EVMYulLean) | ERC-20 selectors = evm-abi-lean's keccak, `native_decide`d in-build |
| `AbiCrossval.abiLean_transfer_decodes` / `abiLean_dynarray_sum` / `abiLean_mixed_sum` (EVMYulLean) | decode-equivalence for static, dynamic-array, and mixed calldata families |
| `AbiEndToEnd.genABIFrontEnd_entry_returns` (EVMYulLean) | contract entry ‚Üí dispatch ‚Üí ABI-encoded return, end-to-end |
| `AbiBalance.abi_balanceOf_orig_opt_returndata_eq` (EVMYulLean) | **the composed capstone**: a `balanceOf` *call* halts with identical returndata on the original (keccak slot) and the patched (`~addr`) dispatcher, under the per-address storage relation |
| `roundtrip_wf` / `roundtrip_args_wff` (evm-abi-lean) | encode‚Üídecode roundtrip over all well-formed ABI types, nested tuples/structs included ‚Äî the layouts `venom_opt.abi` and the checked-in vectors implement |

All pillars are drift-guarded by `tests/test_evmyullean_mapping.py`
(`make check-mapping`), which also reports the EVMYulLean / evm-abi-lean
commits validated against.

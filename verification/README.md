# Verification

The peephole replaces the `balanceOf` slot function — a keccak hash — with the
bitwise complement `~addr`. That is sound **iff the replacement is injective**;
the theorems below establish it.

## In this repo (self-contained, mathlib-free)

[`VenomOpt/Peephole.lean`](VenomOpt/Peephole.lean) proves the mathematical core
on `BitVec 256` (standard axioms only):

| theorem | statement |
|---|---|
| `optSlot_injective` | `~·` is injective (it is its own inverse) |
| `distinct_addresses_distinct_opt_slots` | `a ≠ b → ~a ≠ ~b` — no slot collisions |
| `write_no_alias` | writing holder `a`'s balance never changes holder `b`'s (`b ≠ a`) |
| `read_own_write` | a holder reads back exactly what was written to their slot |
| `unpatched_read_misses_patched_write` | **completeness matters**: a write via `~key` is invisible to a read still deriving the slot any other way — a *partial* patch splits the map, so the patcher must rewrite every site (the differential tests are the completeness oracle) |

Build & check:

```bash
cd verification && lake build      # builds in seconds — no mathlib
```

## The full, end-to-end statement (EVMYulLean)

The proof here is the *slot-function* argument. That the **rewritten EVM
bytecode** computes the same storage as the original — against EVMYulLean's real
`EVM.step` — is proved in the EVMYulLean development:

| EVMYulLean result | what it adds |
|---|---|
| `NoAlias.distinct_addresses_distinct_opt_slots` | the same injectivity, on `UInt256.lnot` |
| `SlotAbstraction.realizes_write_opt` | the `~addr` storage scheme realizes balance write-through **unconditionally** |
| `SlotAbstraction.write_opt_preserves_named` | a `~addr` write never disturbs a named low slot (`totalSupply`, …) |
| `BalanceSlot.venomBalanceLoad_orig_opt_equiv` | original (keccak) and optimized (`~addr`) loads agree under the storage relation |
| `Solvency.transfer_preserves_solvent` / `Erc20.*` | the rewrite preserves `Σ balances = totalSupply` and the ERC-20 spec |

The rewrite operates on Venom IR, which is then compiled to EVM. That the
compiler step itself is sound — the **generated EVM program simulates the
source Venom function** — is proved in EVMYulLean's Venom→EVM codegen
development, so the compiled output of the (optimized) Venom is correct too:

| EVMYulLean result (`EvmYul/Venom/Hol/Codegen/`) | what it adds |
|---|---|
| `CodegenCorrectness.codegen_fn_correct` / `codegen_correct` | the generated program simulates the source Venom function / context |
| `CodegenCorrectness.codegenFuel_correct_nonvacuous_witness` | …non-vacuously — a real generated program, not an empty/vacuous witness |
| `GenBlockSim/CfgHfsim.genBlockSimulation` | the per-block simulation capstone the above rests on |
| `GenInstSim/JoinProducers.genRegularInstPlan_log_sim` | end-to-end sim for a variable-arity instruction (LOG), incl. the fixed operand order |

The differential harness drives **dynamic-ABI** calldata (DynArray / Bytes /
struct) generated by evm-abi-lean's verified encoder. That those byte layouts
are the right ones — and that the rewrite is invisible at the ABI-call level —
is pinned by two more maps:

| result | what it adds |
|---|---|
| `AbiCrossval.erc20_selectors_match_keccak` (EVMYulLean) | ERC-20 selectors = evm-abi-lean's keccak, checked in-build by `decide +kernel` — on **base axioms**, not `native_decide` |
| `AbiCrossval.abiLean_transfer_decodes` / `abiLean_dynarray_sum` / `abiLean_mixed_sum` (EVMYulLean) | decode-equivalence for static, dynamic-array, and mixed calldata families |
| `AbiEndToEnd.genABIFrontEnd_entry_returns` (EVMYulLean) | contract entry → dispatch → ABI-encoded return, end-to-end |
| `AbiBalance.abi_balanceOf_orig_opt_returndata_eq` (EVMYulLean) | **the composed capstone**: a `balanceOf` *call* halts with identical returndata on the original (keccak slot) and the patched (`~addr`) dispatcher, under the per-address storage relation |
| `Codec.roundtrip` / `roundtrip_args` (evm-abi-lean) | encode→decode roundtrip over all valid ABI types, nested tuples/structs included — the layouts `venom_opt.abi` and the checked-in vectors implement. (Formerly `roundtrip_wf`/`roundtrip_args_wff` in `EvmAbi/Roundtrip.lean`; the Ty-indexed rewrite refined values by their type index, so the well-formedness hypotheses — and the suffixes — went away.) |
| `CodecEval.encodeF_eq_encode` / `encodeArgs_eq_encodeF` (evm-abi-lean) | the kernel-reducible encoder mirror + bridge. `encode`/`decode` are well-founded-recursive over `Ty`, hence opaque to the kernel; this bridge is what lets the `AbiCrossval` checks above evaluate concrete calldata on base axioms instead of falling back to `native_decide` |

All pillars are drift-guarded by `tests/test_evmyullean_mapping.py`
(`make check-mapping`), which also reports the EVMYulLean / evm-abi-lean
commits validated against.
